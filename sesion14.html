¿Por Qué Probamos el Software?

Más Allá de "Encontrar Bugs"

Las pruebas de software son una investigación sistemática para evaluar la calidad de una pieza de software y reducir el riesgo de fallos en operación. No se trata solo de encontrar errores, sino de:

Verificar que el software cumple los requisitos

¿Hace lo que se supone que debe hacer?

Validar que el software satisface las necesidades del usuario

¿Resuelve el problema real del cliente?

Aumentar la confianza

Proporcionar a los interesados (clientes, directivos) un nivel de confianza en que el producto funcionará como se espera.

Prevenir defectos

La preparación de las pruebas en etapas tempranas ayuda a identificar omisiones o ambigüedades en los requisitos y el diseño.

Reducir el riesgo

Identificar fallos potenciales antes de que causen daños financieros, de reputación o incluso físicos en el mundo real.

🏗️

La Filosofía del Testing

Probar es una actividad constructiva, no destructiva. Su objetivo final es mejorar la calidad.

⚖️

Los 7 Principios del Testing (ISTQB)

Las Leyes Universales de las Pruebas

ISTQB - International Software Testing Qualifications Board

La comunidad internacional de testing (ISTQB) ha definido siete principios que son la base de toda prueba profesional:

1

Las pruebas demuestran la presencia de defectos, no su ausencia

No podemos probar que un software no tiene errores, solo que los errores que hemos buscado no están presentes.

2

Las pruebas exhaustivas (probar todo) son imposibles

Probar todas las combinaciones de entradas y condiciones es inviable, excepto en casos triviales. Debemos usar técnicas y análisis de riesgo para enfocar el esfuerzo.

3

Las pruebas tempranas ahorran tiempo y dinero

Iniciar las pruebas lo antes posible en el ciclo de vida reduce drásticamente el costo de la corrección.

4

Los defectos se agrupan (Principio de Pareto)

El 80% de los problemas se suelen encontrar en el 20% de los módulos del sistema.

5

La paradoja del pesticida

Si ejecutas las mismas pruebas una y otra vez, eventualmente dejarán de encontrar nuevos defectos. Los casos de prueba deben revisarse y actualizarse regularmente.

6

Las pruebas dependen del contexto

La forma en que se prueba un sitio de e-commerce es diferente a cómo se prueba el software de un marcapasos. El riesgo, la tecnología y el uso definen la estrategia.

7

La falacia de "ausencia de errores"

Encontrar y corregir muchos defectos no servirá de nada si el sistema construido es inútil y no satisface las necesidades del usuario.

Los Fundamentos del Testing Profesional

Estos principios son la brújula que guía a todo profesional del testing para tomar decisiones informadas y efectivas.

Niveles de Pruebas (La Pirámide)

Construyendo Confianza Nivel por Nivel

Las pruebas se organizan en niveles, comenzando desde las partes más pequeñas hasta el sistema completo.

1

Nivel 1: Pruebas Unitarias (Unit Tests)

¿Qué prueban?

Unidades individuales de código (una función, un método, una clase) de forma aislada.

¿Quién las hace?

El desarrollador que escribe el código.

Objetivo:

Asegurar que cada pieza del rompecabezas funciona correctamente por sí sola.

2

Nivel 2: Pruebas de Integración (Integration Tests)

¿Qué prueban?

Cómo interactúan dos o más unidades/componentes entre sí.

¿Quién las hace?

Desarrolladores o Testers de QA.

Objetivo:

Asegurar que las piezas del rompecabezas encajan y se comunican correctamente.

3

Nivel 3: Pruebas de Sistema (System Tests)

¿Qué prueban?

El sistema completo, ya integrado, contra los requisitos funcionales y no funcionales.

¿Quién las hace?

El equipo de QA, de forma independiente al desarrollo.

Objetivo:

Asegurar que el rompecabezas completo se ve y funciona como el modelo de la caja.

4

Nivel 4: Pruebas de Aceptación (UAT - User Acceptance Tests)

¿Qué prueban?

Si el sistema es "apto para su propósito" desde la perspectiva del cliente o usuario final.

¿Quién las hace?

El cliente, los usuarios finales o un representante.

Objetivo:

Obtener la validación final y la luz verde para el lanzamiento.

🏗️

La Pirámide de Testing

Cada nivel construye sobre el anterior, creando una base sólida de confianza en la calidad del software.

🌈

Tipos de Pruebas (El Espectro)

Funcionales vs. No Funcionales

Podemos clasificar las pruebas en dos grandes familias, según lo que evalúan.

Pruebas Funcionales

Enfoque:

"¿El sistema hace lo correcto?"

Verifican:

Las funciones y características del software. Se basan en los requisitos y especificaciones.

Ejemplos:

Pruebas de humo (Smoke Testing)
Pruebas de regresión (Regression Testing)
Pruebas de interfaz de usuario (UI Testing)

Pruebas No Funcionales

Enfoque:

"¿El sistema lo hace de la forma correcta?"

Verifican:

Los atributos de calidad del software (el "cómo").

Ejemplos:

Pruebas de Rendimiento: ¿Es rápido y escalable? (Pruebas de carga, estrés).
Pruebas de Usabilidad: ¿Es fácil e intuitivo de usar?
Pruebas de Seguridad: ¿Es vulnerable a ataques?
Pruebas de Compatibilidad: ¿Funciona en diferentes navegadores, sistemas operativos o dispositivos?
⚖️

El Equilibrio Perfecto

Un software de calidad necesita ambos tipos de pruebas: debe hacer lo correcto Y hacerlo de la forma correcta.

Laboratorio 3 - "Diseño de Casos de Prueba"

Pensando como un Tester: Rompiendo un Login

🎯 Objetivo

Aplicar el pensamiento analítico para diseñar casos de prueba efectivos que cubran escenarios positivos, negativos y límite para una funcionalidad común.

Funcionalidad a Probar

Una pantalla de inicio de sesión (Login) con dos campos: "Usuario" y "Contraseña", y un botón "Ingresar".

Reglas de Negocio

• El usuario debe tener entre 6 y 20 caracteres.

• La contraseña debe tener al menos 8 caracteres, una mayúscula y un número.

Instrucciones

1. En equipos, diseñen al menos 10 casos de prueba para esta funcionalidad.

2. Usen una tabla simple con las columnas: ID del Caso, Descripción, Pasos a Seguir, Dato de Prueba, Resultado Esperado.

3. Asegúrense de incluir:
Casos Positivos: El "camino feliz" (datos válidos).
Casos Negativos: Intentos con datos inválidos, campos vacíos, etc.
Casos Límite: Pruebas en los bordes de las reglas (ej. un usuario de 5, 6, 20 y 21 caracteres).

Entregable

La tabla con los casos de prueba diseñados.

Pensamiento de Tester

Un buen tester no solo piensa en lo que debería funcionar, sino en todas las formas creativas en que algo podría fallar.
