🏗️ Los 4 Niveles Clásicos de Pruebas

De lo pequeño a lo grande: una estrategia integral

🎯 "Cada nivel tiene su propósito específico"

Desde componentes individuales hasta la aceptación del usuario final

🎯 Objetivo: Comprender los 4 niveles de prueba y cuándo aplicar cada uno

🔬 Pruebas Unitarias

📋 Características Principales

  • Validan componentes individuales como funciones, métodos o clases
  • Normalmente automatizadas y ejecutadas frecuentemente
  • Rápidas de ejecutar - segundos o milisegundos
  • Aisladas - no dependen de otros componentes externos

🛠️ Herramientas Populares

☕ JUnit (Java)

Framework estándar para pruebas unitarias en Java, con anotaciones simples y assertions potentes

🐍 PyTest (Python)

Framework flexible para Python con sintaxis simple y funcionalidades avanzadas de testing

🟨 Jest (JavaScript)

Framework completo para JavaScript con mocking, coverage y testing de componentes

🔷 NUnit (.NET)

Framework robusto para aplicaciones .NET con amplio soporte para diferentes tipos de tests

⚡ Ventaja Clave: Detección temprana de errores con feedback inmediato

🔗 Pruebas de Integración

📋 Propósito Principal

  • Verifican la interacción entre módulos o componentes
  • Detectan problemas en las interfaces entre sistemas
  • Validan el flujo de datos entre diferentes partes del sistema
  • Identifican incompatibilidades entre componentes integrados

🔄 Enfoques de Integración

1
💥 Big Bang

Integrar todos los módulos simultáneamente - rápido pero difícil de debuggear

2
📈 Incremental

Integrar módulos uno por uno - más controlado y fácil de localizar errores

3
⬇️ Top-Down

Comenzar desde módulos de alto nivel hacia abajo - requiere stubs

4
⬆️ Bottom-Up

Comenzar desde módulos de bajo nivel hacia arriba - requiere drivers

🎯 Clave: Los errores de integración son diferentes a los errores unitarios

🖥️ Pruebas de Sistema

📋 Alcance Completo

  • Evalúan el sistema completo como una unidad integrada
  • Requisitos funcionales - ¿hace lo que debe hacer?
  • Requisitos no funcionales - ¿qué tan bien lo hace?
  • Ambiente de producción o lo más similar posible
⚡ Rendimiento

Tiempo de respuesta, throughput, uso de recursos bajo diferentes cargas de trabajo

🔒 Seguridad

Vulnerabilidades, autenticación, autorización, protección de datos sensibles

👤 Usabilidad

Facilidad de uso, navegación intuitiva, experiencia del usuario final

🔄 Confiabilidad

Disponibilidad, recuperación ante fallos, estabilidad a largo plazo

🎯 Perspectiva del Usuario Final

Las pruebas de sistema validan que el producto completo funciona según las expectativas del usuario en condiciones reales

🌐 Ambiente: Debe replicar las condiciones de producción lo más fielmente posible

✅ Pruebas de Aceptación

📋 El Veredicto Final

  • Realizadas por el cliente o usuario final real
  • Validan expectativas del negocio más que especificaciones técnicas
  • Determinan si el producto está listo para ser entregado
  • Enfoque en valor de negocio y satisfacción del usuario

👥 Aceptación del Usuario (UAT)

Características:
  • Usuarios reales del sistema
  • Escenarios de uso cotidiano
  • Validación de workflows
  • Feedback cualitativo

💼 Aceptación del Negocio (BAT)

Características:
  • Stakeholders de negocio
  • Criterios de éxito definidos
  • ROI y métricas de negocio
  • Decisión go/no-go

Análisis de Falla Histórica

💥 El Desastre de 1996

📖 Contexto Histórico

El cohete Ariane 5 explotó 37 segundos después del lanzamiento debido a un error de conversión numérica en el software de navegación. Pérdida: $500 millones USD

🔍 Detalles Técnicos

Un valor de 64 bits (velocidad horizontal) se intentó convertir a 16 bits, causando overflow. El sistema interpretó esto como una falla crítica y activó la autodestrucción

💭 Pregunta de Análisis

¿En qué nivel de prueba pudo haberse detectado este error de conversión numérica?

🤔 Opciones para Considerar

🔬
Pruebas Unitarias

¿La función de conversión individual habría mostrado el overflow?

🔗
Pruebas de Integración

¿La interacción entre módulos habría revelado el problema?

🖥️
Pruebas de Sistema

¿Simulaciones completas con datos reales habrían detectado el error?

✅
Pruebas de Aceptación

¿Los usuarios finales habrían identificado el comportamiento anómalo?
